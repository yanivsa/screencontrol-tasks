# תוכנית פרויקט סופית: משימות תמורת דקות מסך

## 1. מטרת הפרויקט

לבנות אפליקציה משפחתית עבור אורי ואיתן שמחליפה את רעיון המטבעות של FinanKids בשיטה ברורה של "משימות תמורת דקות מסך".

הזרימה המרכזית:

1. ההורה יוצר משימות קבועות או משתנות.
2. כל משימה שווה כמות דקות מסך.
3. הילד מסמן שהמשימה בוצעה.
4. אם צריך, הילד מצרף תמונה כהוכחה.
5. ההורה מקבל בקשת אישור.
6. ההורה מאשר, דוחה, או משנה את כמות הדקות.
7. באישור, הדקות נכנסות לארנק של הילד.
8. כשההורה נותן בפועל זמן מסך דרך Family Link, טלוויזיה, סוני או מכשיר אחר, ההורה מפחית את הדקות מהארנק.
9. הילד יכול לבקש לנצל דקות, אבל ההורה הוא זה שמאשר ומוריד את הדקות.

העיקרון: להפוך זמן מסך ממאבק יומי לשיטה עקבית, מדידה והוגנת.

## 2. מה למדנו מ-FinanKids ומה משנים

מה-XAPK המקורי `com.finankidz.finankids_1.2.2.xapk` עולה שהאפליקציה המקורית היא Flutter WebView שמטעין את `https://finankidz.com`, עם OneSignal, Firebase Messaging, RevenueCat ו-Bubble בצד השרת.

מסקנה: לא עורכים את ה-APK ולא בונים עליו. בונים מוצר חדש, נקי, שמאמץ את הרעיונות הטובים:

- אזור הורה ואזור ילד.
- משימות שהילד מגיש לאישור.
- תגמול אחרי אישור הורה.
- ארנק אישי לכל ילד.
- חוויה משחקית חיובית.
- הוכחת ביצוע בתמונה.
- התראות להורה ולילד.

מה לא משמרים בגרסה הראשונה:

- מטבעות.
- יהלומים.
- XP.
- לבבות.
- מנויים.
- RevenueCat.
- Bubble.
- WebView כממשק ראשי.

המטבע היחיד בגרסה הראשונה: דקות מסך.

## 3. החלטת ארכיטקטורה

הגרסה הראשונה צריכה לתמוך בכמה מכשירים:

- הילד מסמן משימה מהמכשיר שלו.
- ההורה מאשר מהטלפון שלו.
- היתרה מתעדכנת לכל המכשירים.
- אם אין אינטרנט, הפעולה נשמרת מקומית ומסתנכרנת כשחוזרים לרשת.

לכן לא מספיק SQLite מקומי בלבד. SQLite יישאר רק כ-cache מקומי ותור סנכרון, לא כמקור האמת.

### Backend

נשתמש ב-Cloudflare:

- Cloudflare Workers: API מרכזי לאפליקציה.
- Cloudflare D1: מסד נתונים SQL מנוהל, מבוסס SQLite.
- Cloudflare R2: אחסון תמונות הוכחת ביצוע.
- Cloudflare GitHub integration או GitHub Actions: פריסה אוטומטית אחרי push.

הערה חשובה: השם הנכון הוא D1 למסד נתונים ו-R2 לאחסון קבצים. אין צורך ב-"R1".

האפליקציה לא תדבר ישירות עם D1 או R2. היא תדבר עם Worker API בלבד. זה הכרחי כדי לא לחשוף מפתחות, לאפשר הרשאות, ולשמור על API יציב.

### Frontend

אפליקציית Flutter Native:

- Android-first.
- iOS בהמשך.
- עברית RTL.
- UI מקומי, לא WebView.
- SQLite מקומי ל-cache ולפעולות שלא סונכרנו.

### עדכונים בלי לעדכן אפליקציה

אפשר לעדכן הרבה דברים דרך Cloudflare בלי לעדכן את האפליקציה:

- משימות מוכנות.
- חוקים משפחתיים.
- תקרות דקות.
- טקסטים.
- הגדרות Gamification.
- שינויי API פנימיים, כל עוד החוזה מול האפליקציה נשמר.

אבל שינויי UI גדולים באפליקציה עדיין ידרשו עדכון אפליקציה. לכן נבנה API יציב ו-Remote Config כדי למזער עדכונים.

## 4. משתמשים והרשאות

### Onboarding ורישום ראשוני

בפעם הראשונה שהאפליקציה נפתחת, ההורה יוצר משפחה:

- מזהה הורה/אימייל: `yanivsa`
- PIN ראשוני להורה: `2602`
- שמות ילדים: אורי ואיתן
- PIN נפרד לכל ילד

הערת אבטחה: ה-PIN `2602` לא יישמר כטקסט גלוי במסד הנתונים או בקוד. הוא יישמר רק כ-hash בצד השרת. אם בעתיד משתמשים באימייל אמיתי לשחזור, צריך לעדכן את `yanivsa` לכתובת מלאה.

API נדרש:

- `POST /families/register`
- `POST /auth/parent-login`
- `POST /auth/child-login`
- `POST /auth/recover-parent`

שחזור PIN הורה:

- ב-MVP: שחזור דרך קוד חד-פעמי שנשלח לאימייל ההורה, אם מוגדר אימייל מלא.
- אם אין אימייל מלא, שחזור יתבצע רק דרך מסך ניהול מקומי/seed admin בזמן פיתוח.

### הורה

ההורה יכול:

- ליצור משימות.
- לערוך משימות.
- למחוק או להשהות משימות.
- לאשר או לדחות ביצוע.
- לשנות דקות לפני אישור.
- להוסיף או להפחית דקות ידנית.
- לראות היסטוריה מלאה.
- להגדיר PIN לילדים.
- להגדיר האם מותר חוב.
- להגדיר תקרות יומיות ושבועיות.

### אורי

ילד עם:

- פרופיל אישי.
- PIN אישי.
- ארנק דקות.
- משימות פתוחות.
- משימות קבועות.
- משימות משתנות.
- היסטוריית דקות.
- בקשת ניצול דקות מסך לאישור הורה.

### איתן

כמו אורי, עם פרופיל ו-PIN נפרדים.

### PIN

לכל אחד יש קוד נפרד:

- קוד הורה.
- קוד אורי.
- קוד איתן.

אפשר לממש PIN כ-4 ספרות או כ-4 צבעים/צורות. ב-MVP נבחר 4 ספרות, כי זה פשוט יותר לבנייה ולבדיקה.

ילד לא יכול:

- להיכנס לפרופיל של ילד אחר בלי PIN.
- לאשר לעצמו משימות.
- להוסיף לעצמו דקות.
- למחוק היסטוריה.

## 5. מודל משימות: קבועות ומשתנות

כדי לתמוך נכון במשימות קבועות ומשתנות, מפרידים בין "תבנית משימה" לבין "מופע משימה".

### TaskTemplate

תבנית שמגדירה משימה שחוזרת או משימה מוכנה לשימוש.

שדות:

- `id`
- `familyId`
- `title`
- `description`
- `defaultRewardMinutes`
- `assignedChildIds`
- `requiresPhoto`
- `isActive`
- `scheduleType`: `one_time`, `daily`, `weekly`, `custom`
- `daysOfWeek`: למשל ראשון/שלישי/חמישי
- `timeWindowStart`: אופציונלי
- `timeWindowEnd`: אופציונלי
- `createdByParentId`
- `createdAt`
- `updatedAt`

דוגמאות:

- "לסדר מיטה" - יומי - 5 דקות.
- "לקרוא 20 דקות" - יומי - 15 דקות.
- "לפנות מדיח" - לפי צורך - 10 דקות.
- "סידור חדר ביום שישי" - שבועי - 30 דקות.

### TaskInstance

מופע בפועל של משימה לילד בתאריך מסוים.

שדות:

- `id`
- `templateId`
- `familyId`
- `childId`
- `title`
- `description`
- `rewardMinutes`
- `requiresPhoto`
- `status`: `open`, `submitted`, `approved`, `rejected`, `expired`, `cancelled`
- `dueAt`
- `createdAt`
- `submittedAt`
- `reviewedAt`
- `reviewedByParentId`

למה צריך מופעים:

- אם משימה יומית חוזרת, כל יום צריך סטטוס משלו.
- אם אורי ביצע ואיתן לא, לכל ילד יש מופע נפרד.
- אם ההורה משנה תגמול רק להיום, זה לא משנה את כל התבנית.

### משימה משתנה

משימה משתנה היא TaskInstance בלי תבנית חוזרת, או TaskTemplate מסוג `one_time`.

דוגמאות:

- "לעזור להכניס קניות עכשיו" - 20 דקות.
- "לסדר את המשחקים בסלון" - 15 דקות.
- "להתארגן מהר כי יוצאים" - 10 דקות.

### משימה שהילד מציע

כמו ב-FinanKids, כדאי לאפשר לילד להציע משימה:

- הילד כותב/בוחר רעיון.
- מציע כמה דקות הוא מבקש.
- ההורה מאשר/דוחה/משנה.
- אם ההורה מאשר, נוצרת TaskInstance פתוחה או מאושרת לפי החלטת ההורה.

## 6. שיטת הדקות

### ארנק דקות

לכל ילד יש:

- `availableMinutes`
- `pendingMinutes`
- `earnedToday`
- `earnedThisWeek`
- `spentToday`
- `spentThisWeek`
- `debtMinutes`

הערה: את רוב הערכים אפשר לחשב מהיסטוריית תנועות, אבל נשמור יתרה מחושבת בטבלת child כדי שהאפליקציה תהיה מהירה.

### צבירת דקות

דקות נכנסות לארנק רק באישור הורה.

תהליך:

1. הילד מגיש משימה.
2. נוצר `TaskSubmission`.
3. הסטטוס עובר ל-`submitted`.
4. ההורה מאשר.
5. נוצר `MinuteTransaction` מסוג `earn`.
6. יתרת הילד מתעדכנת.
7. הילד מקבל חיווי חיובי.

### ניצול דקות: בקשת ילד ואישור הורה

ברירת המחדל: הילד לא "לוקח" זמן מסך לבד. הילד מבקש לנצל זמן, והבקשה מגיעה ליוזר של ההורה.

הילד בוחר:

- 5 דקות
- 10 דקות
- 15 דקות
- 30 דקות
- מותאם אישית

נוצרת `ScreenTimeRequest` בסטטוס `pending`.

ההורה יכול:

- לאשר את הבקשה.
- לדחות את הבקשה.
- לשנות את כמות הדקות ולאשר.
- לאשר אוטומטית לפי חוק משפחתי, למשל "עד 30 דקות ביום בלי אישור נוסף".

ברגע שההורה מאשר, הדקות יורדות מהארנק ונרשם `ScreenUsageLog`. אין טיימר בתוך האפליקציה.

המערכת בודקת לפני אישור:

- האם יש מספיק יתרה.
- האם הילד לא עבר תקרת שימוש יומית.
- אם היתרה שלילית, האם ההורה בכל זאת מאשר חריגה.

### ניצול זמן מחוץ לאפליקציה

לא כל זמן מסך יקרה בתוך האפליקציה. חלק מהזמן יכול להיות:

- Family Link בטלפון/טאבלט.
- טלוויזיה.
- סוני / PlayStation.
- מחשב.
- טאבלט שלא מריץ את האפליקציה.

לכן האפליקציה היא "ארנק והרשאות", לא בהכרח כלי הנעילה עצמו.

להורה יהיו שתי פעולות:

- `אשר שימוש`: הילד ביקש, ההורה מאשר, ואז ההורה נותן בפועל זמן דרך Family Link / טלוויזיה / סוני.
- `רשום ניצול ידני`: ההורה מעדכן בדיעבד שאורי/איתן ניצל X דקות בטלוויזיה/סוני/Family Link.

כל ניצול זמן כולל מקור:

- `family_link`
- `tv`
- `playstation`
- `computer`
- `tablet`
- `manual_parent`
- `other`

כאשר שימוש מאושר או נרשם ידנית:

- נוצר `ScreenUsageLog`.
- נוצר `MinuteTransaction` מסוג `spend`.
- הדקות יורדות מיד מהיתרה.

## 7. התראות

ב-MVP יהיו התראות בתוך האפליקציה ו-polling כל 30 שניות כשהאפליקציה פתוחה. Push notifications אמיתיות דרך FCM יתווספו בשלב 2.

### התראות להורה

ההורה יקבל התראה כאשר:

- ילד הגיש משימה לאישור.
- ילד ביקש לנצל דקות מסך.
- ילד הציע משימה חדשה.
- ילד העלה תמונה למשימה.
- יש משימה יומית שלא בוצעה עד שעה מוגדרת, למשל 17:00.

### התראות לילד

הילד יקבל התראה כאשר:

- ההורה אישר משימה.
- ההורה דחה משימה.
- ההורה אישר בקשת זמן מסך.
- ההורה דחה בקשת זמן מסך.
- ההורה הוסיף או הפחית דקות.
- יש תזכורת למשימה יומית שלא בוצעה.

### מודל טכני

- שלב 1: `GET /notifications?since=cursor` ב-polling.
- שלב 2: FCM push notifications.
- כל התראה נשמרת ב-D1 כדי שלא תאבד אם האפליקציה סגורה.
- התראות רגישות לא מציגות תוכן מלא במסך נעילה, רק "יש בקשה חדשה".

## 8. ניצול זמן ללא טיימר

החלטה סופית: אין טיימר באפליקציה ב-MVP.

הסיבה:

- זמן המסך בפועל ניתן מחוץ לאפליקציה: Family Link, טלוויזיה, סוני, מחשב או מכשיר אחר.
- ההורה הוא זה שמחליט מתי הילד קיבל מסך וכמה דקות להפחית.
- טיימר פנימי יוצר מורכבות מיותרת ולא באמת שולט בטלוויזיה/סוני/Family Link.

שני מסלולי ניצול:

1. בקשת ילד:
   - הילד מבקש 15/30/60 דקות ומציין מקור: טלוויזיה, סוני, טאבלט וכו'.
   - ההורה מקבל בקשה.
   - ההורה מאשר, משנה כמות, או דוחה.
   - באישור, הדקות יורדות מיד מהארנק.

2. רישום הורה:
   - ההורה נתן בפועל זמן מסך.
   - ההורה פותח "רישום ניצול".
   - בוחר ילד, מקור, כמות דקות וסיבה.
   - הדקות יורדות מיד מהארנק.

אין החזרת יתרה אוטומטית, כי אין מדידת זמן בפועל בתוך האפליקציה. אם ההורה רוצה להחזיר דקות, הוא מבצע תיקון ידני מסוג `earn` או `refund`.

## 9. יתרה שלילית וחוב

המערכת תתמוך ביתרה שלילית, אבל רק בפעולת הורה מפורשת.

ברירת מחדל:

- הרצפה הרגילה היא 0.
- ילד לא יכול להכניס את עצמו למינוס.
- ילד לא יכול לבקש שימוש אם אין לו מספיק דקות, אלא אם ההורה מאשר חריגה.
- הורה יכול להפחית דקות גם מעבר למינוס 60.
- מינוס 60 הוא רק סף אזהרה חזותי, לא חסימה.

דוגמה:

- לאיתן יש 0 דקות.
- ההורה מפחית 10 דקות בגלל התנהגות.
- איתן נמצא ב-`-10`.
- המשימות הבאות קודם מכסות את החוב.
- אם איתן מרוויח 15 דקות, היתרה החדשה היא 5.

כללי חוב:

- כשהילד במינוס, דקות ממשימות קודם מכסות את החוב.
- אם ההורה מוריד מעבר למינוס 60, האפליקציה מציגה אזהרה אבל מאפשרת.
- כל הפחתה חייבת סיבה.
- היסטוריה מציגה בבירור מי הוריד, כמה, ומדוע.

ההורה יוכל לשנות את סף האזהרה בהגדרות.

## 10. הוכחת ביצוע בתמונה

חלק מהמשימות יכולות לדרוש תמונה.

דוגמאות:

- סידור מיטה.
- סידור חדר.
- הכנת תיק.
- שולחן נקי.

### תהליך

1. ההורה מסמן `requiresPhoto`.
2. הילד לא יכול להגיש בלי לצלם או להעלות תמונה.
3. האפליקציה מבקשת הרשאת מצלמה/גלריה.
4. התמונה עולה ל-Cloudflare R2 דרך Worker.
5. ב-D1 נשמר רק מזהה/נתיב התמונה, לא הקובץ עצמו.
6. ההורה רואה תמונה במסך האישור.

### אחסון

Cloudflare R2 מתאים לזה:

- אחסון אובייקטים.
- מתאים לתמונות.
- נגיש מ-Workers.
- אפשר לשמור תמונות כ-private bucket.

התמונות לא יהיו ציבוריות. Worker יחזיר URL חתום/זמני או יזרים את התמונה רק למשתמש מורשה.

### מדיניות מחיקת תמונות

כדי למנוע הצטברות מיותרת של תמונות:

- כל תמונת הוכחת ביצוע תישמר עד 7 ימים בלבד.
- ההורה יכול למחוק תמונה ידנית לפני סוף השבוע.
- אין מצב "שמירה חשובה" ב-MVP, כדי לא לסבך פרטיות ואחסון.
- אחרי המחיקה, ההיסטוריה נשארת ב-D1, אבל בלי קובץ התמונה.

מימוש:

- R2 Lifecycle rules למחיקה אוטומטית לפי prefix.
- Prefix מומלץ: `families/{familyId}/submissions/{yyyy-mm}/...`
- Worker יעדכן `photo_deleted_at` ב-D1 בסריקת תחזוקה יומית, כדי שהאפליקציה תדע שהתמונה כבר לא זמינה.

### הרשאות

ב-Android צריך:

- הרשאת מצלמה אם מצלמים מתוך האפליקציה.
- הרשאת גלריה/תמונות אם בוחרים תמונה קיימת.

ב-MVP מומלץ להתחיל עם צילום מתוך האפליקציה בלבד כדי לצמצם מורכבות.

## 13. מנגנון משימות קבועות ו-Cron

משימות קבועות לא ייווצרו רק כשהילד פותח את האפליקציה. זה ייצור מצב שההורה לא רואה משימה פתוחה אם הילד לא נכנס.

הפתרון: Cloudflare Cron Trigger.

### התנהגות

כל יום בשעה 06:00 לפי שעון ישראל:

1. Worker scheduled handler רץ.
2. הוא מחפש `task_templates` פעילות.
3. הוא בודק אילו תבניות רלוונטיות לאותו יום.
4. הוא יוצר `task_instances` לכל ילד משויך.
5. הוא לא יוצר כפילויות אם מופע כבר קיים לאותו תאריך.
6. הוא יוצר התראות תזכורת לפי הגדרות המשפחה.

### הגדרת Wrangler

ב-`wrangler.toml`:

```toml
[triggers]
crons = ["0 3 * * *"]
```

הערה: cron של Cloudflare עובד ב-UTC. `03:00 UTC` הוא `06:00` בישראל בזמן שעון קיץ. אם רוצים דיוק מלא לאורך כל השנה, ה-Worker צריך לחשב את היום לפי timezone `Asia/Jerusalem` בזמן הריצה.

### Lazy fallback

גם עם Cron, ה-API צריך לכלול fallback:

- בכל קריאת `GET /tasks`, השרת בודק אם חסרים מופעים להיום.
- אם Cron נכשל, השרת יוצר את המופעים החסרים.

כך לא תלויים רק ב-Cron.

## 16. סנכרון ואופליין

האפליקציה תשמור cache מקומי, אבל מקור האמת הוא Cloudflare D1.

### באינטרנט

- כל פעולה נשלחת מיד ל-Worker.
- Worker כותב ל-D1/R2.
- האפליקציה מקבלת מצב מעודכן.
- מסכים אחרים מקבלים עדכון דרך polling קצר או sync cursor.

### בלי אינטרנט

האפליקציה:

- מאפשרת צפייה במידע האחרון.
- מאפשרת להגיש משימה.
- מאפשרת להגיש משימה וליצור בקשת זמן מסך.
- שומרת פעולה בתור `pending_sync`.

כאשר האינטרנט חוזר:

- האפליקציה שולחת `POST /sync/push`.
- Worker בודק התנגשויות.
- Worker מחזיר מצב עדכני.

### כללי פתרון התנגשויות

- אישור הורה מהשרת גובר על פעולה מקומית.
- לא ניתן לאשר פעמיים אותה משימה.
- אם ילד הגיש אופליין והמשימה כבר פגה, היא עוברת לסטטוס `needs_parent_review`.
- אם בקשת זמן מסך נוצרה אופליין, היא מסונכרנת כבקשה ממתינה; רק ההורה יכול לאשר ולהפחית דקות.

## 20. מקורות טכנולוגיים

- Cloudflare D1: מסד נתונים SQL מנוהל עם סמנטיקת SQLite וגישה דרך Workers.
- Cloudflare R2: אחסון אובייקטים לתמונות וקבצים, עם גישה דרך Workers.
- Cloudflare Workers: שכבת API שמתחברת ל-D1/R2 דרך bindings.
- Cloudflare GitHub integration: deploy אוטומטי מ-GitHub ל-Worker.
- Cloudflare Cron Triggers: יצירת מופעי משימות קבועות לפי לוח זמנים.
- Cloudflare R2 Lifecycle rules: מחיקת תמונות ישנות אוטומטית.

## 22. חיתוך מצב פיתוח

סטטוס נכון לעכשיו: האפליקציה בשלה כמעט לחלוטין ל-Internal Testing ב-Google Play. היא עדיין לא 100% production בגלל חסם חיצוני של R2 שלא הופעל בחשבון Cloudflare ובגלל שלא בוצעה התקנה על מכשיר Android אמיתי דרך Play Console.

הערכת ביצוע כוללת: 99%.

הערכת מוכנות ל-Google Play Store Internal Testing: 95%.

הסיבה לפער שנותר: Cloudflare Pages + Worker + D1 חיים בפרודקשן, ה-Frontend פרוס, ה-Worker פרוס, ויש AAB חתום. תמונות אומתו בפרודקשן דרך fallback ל-D1, אבל R2 עצמו לא אומת כי Cloudflare דורש להפעיל R2 בחשבון דרך ה-Dashboard. בנוסף עדיין חסרה בדיקת התקנה על מכשיר Android אמיתי ופתיחת Internal Testing ב-Play Console.
